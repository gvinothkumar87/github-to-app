## EXTENDVALIDITY Payload Rules

Error 711 fires if `isInTransit` is missing. Errors 712/713 fire if transit fields are sent when `isInTransit = "Y"`.

```json
// When goods are IN MOVEMENT (on the road): isInTransit = "Y"
{
  "ewbNo": 123456789012,
  "vehicleNo": "TN01AB1234",
  "fromPlace": "Chennai",
  "fromState": 33,
  "remainingDistance": 100,
  "transDocNo": "",
  "transDocDate": "",
  "transMode": "1",
  "extnRsnCode": 5,
  "extnRemarks": "Due to Break down",
  "fromPincode": 600001,
  "isInTransit": "Y"
}

// When goods are IN STORAGE (transit warehouse): isInTransit = "N"
{
  "ewbNo": 123456789012,
  "vehicleNo": "TN01AB1234",
  "fromPlace": "Chennai",
  "fromState": 33,
  "remainingDistance": 100,
  "transDocNo": "",
  "transDocDate": "",
  "transMode": "1",
  "extnRsnCode": 5,
  "extnRemarks": "Stored at warehouse",
  "fromPincode": 600001,
  "isInTransit": "N",
  "consignmentStatus": "T",
  "transitType": "W",
  "addressLine1": "Godown 1, Chennai",
  "addressLine2": "",
  "addressLine3": ""
}
```

## Sandbox Response Quirks

The TaxPro GSP sandbox frequently returns data **directly at the JSON root** (unwrapped) instead of the standard `{Status: 1, Data: {...}}` envelope. Always check for both patterns:

```typescript
// Detect success from both wrapped and unwrapped responses:
const ewbNo = result.ewbNo || result.EwbNo || result.Data?.ewbNo;
const isSuccess =
  result.Status === 1 || result.Status === '1' ||
  result.status === 1 || result.status === '1' ||
  !!ewbNo;

// Cancel EWB success (unwrapped):
const cancelDate = result.cancelDate || result.CancelDate || result.Data?.cancelDate;

// Vehicle Update success (unwrapped):
const vehUpdDate = result.vehUpdDate || result.VehUpdDate || result.Data?.vehUpdDate;

// Extend Validity success (unwrapped):
const validUpto = result.validUpto || result.ValidUpto || result.Data?.validUpto;
```
